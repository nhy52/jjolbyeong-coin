# 쫄병코인 (Jjolbyeong Coin) 기획서

> 대장(Boss)이 쫄병(Minion)에게 코인을 지급하고, 쫄병은 그 코인으로 대장마켓에서 상품을 사거나 빗토코인에 투자하는 웹 서비스.
> 최종 수정: 2026-07-14

---

## 1. 서비스 개요

### 1.1 한 줄 소개
칭찬 스티커의 디지털 버전. 대장이 쫄병에게 "코인"을 지급하면 쫄병은 실시간으로 코인이 쌓이는 걸 보고, 그 코인으로 대장이 등록한 상품을 사거나 투자상품(빗토코인)에 투자할 수 있다.

### 1.2 핵심 컨셉
- **코인 = 보상/포인트**. 실제 화폐가 아니라 대장이 무한히 발행할 수 있는 그룹 내 포인트.
- **대장마켓 = 보상 교환소**. 네이버 스마트스토어 느낌으로 상품을 등록하고 진열.
- **빗토코인 = 재미 요소**. 랜덤 확률로 오르내리는 투자상품.

### 1.3 타깃/사용 시나리오 (예시)
- 부모(대장)-자녀(쫄병): 심부름·공부 보상으로 코인 지급, "장난감 쿠폰" 상품 구매
- 사장(대장)-직원(쫄병), 팀장-팀원, 커플/친구 등 소규모 그룹

## 2. 역할 및 권한

### 2.1 역할 정의
| 역할 | 설명 |
|------|------|
| **대장 (Boss)** | 그룹의 관리자. 쫄병 승인, 코인 지급, 상품 등록/관리, 쫄병 정보 조회 |
| **쫄병 (Minion)** | 그룹의 멤버. 코인 수령, 대장마켓 구매, 빗토코인 투자 |

> 영어 명칭 제안: 대장 = **Boss**, 쫄병 = **Crew** 또는 **Minion**. (코드/DB에서는 `boss` / `minion` 사용 예정 — 확정 필요)

### 2.2 그룹(멀티 대장) 구조
- 여러 대장이 각자의 **그룹**을 가진다.
- 하나의 그룹 = 대장 1명 + 쫄병 N명.
- **대장마켓 상품과 코인 잔액은 그룹 단위로 격리**된다.
  - 즉 A그룹 대장이 올린 상품은 B그룹 쫄병에게 보이지 않는다.
- **빗토코인 시세는 전역 공용**(모든 그룹 동일). 단 보유 수량/손익은 사용자별. (8.4 참고)

### 2.3 권한 매트릭스
| 기능 | 대장 | 쫄병 |
|------|:----:|:----:|
| 쫄병 가입 승인/거절 | ✅ | ❌ |
| 쫄병에게 코인 지급/차감 | ✅ | ❌ |
| 쫄병 상세정보 조회 | ✅ | ❌(본인만) |
| 상품 등록/수정/삭제 | ✅ | ❌ |
| 상품 신청(마켓 등록 요청) | ❌ | ✅ |
| 상품 신청 승인/거절 | ✅ | ❌ |
| 대장마켓 구매 | ❌(대장 코인 정책 추후 결정) | ✅ |
| 빗토코인 투자 | ✅(대장도 투자 가능) | ✅ |
| 관리자 페이지 접근 | ✅ | ❌ |

## 3. 인증 / 온보딩 플로우

**방식: 쫄병 자가 회원가입 + 대장 승인**

### 3.1 대장 가입
1. 대장 회원가입 (이메일/아이디 + 비밀번호)
2. 그룹 자동 생성 → **초대 코드**(예: 6자리) 발급
3. 대장은 초대 코드를 쫄병에게 공유

### 3.2 쫄병 가입 (자가가입 → 대장 승인)
1. 쫄병 회원가입 시 대장의 **초대 코드** 입력 → 어느 그룹인지 지정
2. 가입 시 프로필 입력: 이름, 전화번호, 주소, 성별, 나이 등
3. 상태 = `pending(승인 대기)`
4. 대장이 관리자 페이지에서 **승인/거절**
5. 승인 시 상태 = `active`, 코인 지갑 생성(잔액 0)

### 3.3 상태 값
`pending` → `active` → (`suspended` / `left`)

> ＊확인 필요: 초대 코드 방식 OK인지, 아니면 그룹을 코드 없이 목록에서 고르게 할지.

## 4. 화면 구성 (모바일 우선 / Mobile-first)

> 전체 UI는 **모바일 친화적**으로 설계. 하단 탭 네비게이션 + 세로 스크롤 카드 레이아웃 기준. 데스크톱은 중앙 정렬 컨테이너로 대응.

### 4.1 공통
- 상단: 로고 / 현재 코인 잔액(실시간)
- 하단 탭바(쫄병): `홈` · `대장마켓` · `빗토코인` · `내정보`
- 하단 탭바(대장): `홈` · `대장마켓` · `관리자` · `내정보`

### 4.2 쫄병 화면
1. **홈(대시보드)**
   - 큰 코인 잔액 카드 (지급 시 **실시간 애니메이션**으로 증가)
   - 최근 코인 내역(지급/구매/투자) 타임라인
2. **대장마켓**
   - 상품 카드 그리드 (이미지·이름·가격)
   - 상품 상세 → 구매 버튼 (코인 차감 확인 모달)
   - 대장이 상품 올리면 **새로고침 없이** 목록에 실시간 추가
   - `상품신청` 탭 — 갖고 싶은 상품을 대장에게 신청(이름·희망가·사유·참고 이미지/링크) + 내 신청 상태 확인(7.3)
3. **빗토코인**
   - 현재 시세 + 시세 차트(선형)
   - 보유 수량 / 평가금액 / 손익
   - 매수/매도 버튼
4. **내정보**
   - 본인 프로필, 구매 내역, 로그아웃

### 4.3 대장 화면
1. **홈**: 그룹 요약(쫄병 수, 총 발행 코인 등), 승인 대기 알림
2. **대장마켓**: 상품 관리(등록/수정/삭제) · 주문 처리 · `신청` 탭(쫄병 상품 신청 승인/거절, 7.3)
3. **관리자 페이지** (아래 5장)
4. **내정보**

## 5. 관리자 페이지 (대장 전용)

### 5.1 쫄병 관리
- 쫄병 목록 (이름, 코인 잔액, 상태)
- 쫄병 상세: **이름, 전화번호, 주소, 성별, 나이, 코인 잔액, 가입일, 구매/투자 내역**
- 승인 대기 목록 → 승인/거절

### 5.2 코인 지급 (칭찬 스티커)
- 쫄병 선택 → **지급할 코인 수량 입력** → 사유(선택) → 지급
- **지급만 지원, 차감 없음**(1차 확정). 벌점성 차감 정책은 추후 검토.
- 지급 즉시 해당 쫄병 화면에 **SSE로 실시간 반영**

### 5.3 상품 등록/관리 (스마트스토어 느낌)
상품 등록 폼:
- 상품 **이미지**(대표 1장 — 1차 확정. 다중 이미지는 추후 확장)
- 상품 **이름**
- **가격**(코인)
- **상세정보**(설명 텍스트, 추후 리치 에디터 고려)
- 재고 수량(선택) / 판매 상태(판매중·품절·숨김)
- 등록/수정/삭제
- 등록 즉시 그룹 쫄병 대장마켓에 **실시간 추가**

## 6. 코인 시스템

### 6.1 지갑 & 원장(Ledger)
- 쫄병마다 코인 지갑(잔액) 보유.
- 모든 변동은 **거래 원장(transaction ledger)** 에 기록 → 잔액은 원장 합계로 검증 가능(감사/디버깅 용이).
- 거래 타입: `GRANT`(대장 지급), `DEDUCT`(차감), `PURCHASE`(상품 구매), `INVEST_BUY`, `INVEST_SELL`.

### 6.2 정합성 원칙
- 구매/투자 시 **서버에서 잔액 검증** 후 트랜잭션 처리(동시성 대비 DB 트랜잭션/행 잠금).
- 잔액 부족 시 구매·투자 거부.

## 7. 대장마켓 & 구매

### 7.1 구매 플로우
1. 쫄병이 상품 선택 → 구매 확인 모달(가격/잔액 표시)
2. 서버: 잔액 검증 → 차감 → `PURCHASE` 원장 기록 → 주문 생성
3. 구매 완료 후 잔액 실시간 갱신

### 7.2 주문/교환 처리 (수령완료 상태관리 — 확정)
- 구매 시 주문 생성 → 상태 `purchased(구매됨)`.
- 대장이 관리자 페이지에서 실제 보상을 지급하면 **`fulfilled(수령완료)`** 로 변경.
- 예: 자녀가 "아이스크림 쿠폰" 구매 → 부모가 실제로 사주면 "수령완료" 체크.
- 주문 상태: `purchased → fulfilled` (필요 시 `canceled`).
- 대장 관리자 페이지에 **주문/구매 목록** 화면 추가(상태별 필터, 수령완료 처리 버튼).
- 신규 주문 발생 시 대장에게 **SSE 알림**(`order.created`).

### 7.3 상품 신청 (쫄병 → 대장) — 확정

쫄병이 "이 상품을 마켓에 올려주세요"라고 대장에게 **신청**하고, 대장이 검토해 **승인/거절**하는 흐름.

**신청 항목**
- 상품 **이름**(필수)
- **희망 가격**(코인, 필수) — 쫄병이 제안, 대장이 최종 조정
- **신청 사유/설명**(선택)
- **참고 이미지**(선택 — 기존 이미지 업로드 재사용)
- **참고 링크(URL)**(선택)

**처리 흐름**
1. 쫄병이 대장마켓 `상품신청` 탭에서 신청 → 상태 `pending`, 그룹 대장에게 **SSE**(`request.created`).
2. 대장이 대장마켓 `신청` 탭에서 검토.
   - **승인**: 신청 내용(이름·희망가격·설명·이미지)으로 **상품을 마켓에 자동 등록**(재고 무제한·상태 `on_sale`). 신청은 `approved`로 바뀌고 생성된 상품(`product_id`)이 연결됨. 상품 등록 이벤트(`product.created`)가 그룹 쫄병에게 브로드캐스트되어 상점에 즉시 노출.
   - **거절**: 신청은 `rejected`, **거절 사유**(선택) 기록.
3. 승인/거절 결과는 신청자·대장에게 **SSE**(`request.updated`)로 반영.
4. 이미 처리된(pending이 아닌) 신청을 다시 승인/거절하면 거부(409).

> 신청 상태: `pending → approved | rejected`. 승인 시 재고·노출 세부 조정은 대장이 등록 후 **상품 수정**으로 처리한다(스마트스토어의 판매자 등록 흐름과 동일 톤).

## 8. 빗토코인 (투자상품)

### 8.1 컨셉
랜덤 확률로 오르내리는 가상 투자상품. 쫄병이 코인으로 매수/매도하며 차익을 노림.

### 8.2 시세 갱신 방식
**서버가 주기적으로 시세 갱신** (선택하신 방식)
- 서버 백그라운드 작업이 **5초마다**(확정) 다음 가격 계산.
- 갱신된 시세를 **SSE로 전 사용자에게 브로드캐스트** → 차트가 실시간으로 움직임.
- 실제 코인처럼 "시세 차트"가 존재.

### 8.3 가격 변동 로직(초안)
- 매 틱마다 랜덤 변동률 적용: `new = old * (1 + r)`, `r`은 예컨대 `-5% ~ +5%` 균등/정규분포.
- 하한가(최소값) 설정으로 0 이하 방지.
- 파라미터(변동폭, 주기, 상승/하락 확률)는 설정값으로 분리해 튜닝 가능하게.

### 8.4 시세 범위 (전역 공용 — 확정)
- **전역 공용 시세 1종**. 모든 그룹이 동일한 빗토코인 차트를 본다.
- 서버 백그라운드 작업 1개가 5초마다 시세를 갱신하고 전체에 브로드캐스트 → 구현 단순, 서버 부하 최소.
- 단, **보유 수량·투자 손익은 사용자별**로 각자 관리(시세만 공용).

### 8.5 투자 처리
- 매수: 코인 차감 → 빗토코인 수량 증가(현재가 기준)
- 매도: 빗토코인 수량 감소 → 코인 증가(현재가 기준)
- 보유 수량/평단가/평가손익 계산.

## 9. 실시간 반영 아키텍처 (SSE)

> 선택하신 **SSE(Server-Sent Events)** 로 구현. 폴링 부하 없이 서버→클라이언트 단방향 푸시.

### 9.1 왜 SSE인가
- 이 서비스의 실시간 요구는 전부 **서버→클라이언트 단방향**(코인 지급, 상품 추가, 시세 갱신). 쫄병이 서버로 실시간 밀어넣을 일은 없음(구매/투자는 일반 REST면 충분).
- HTTP 기반이라 FastAPI에 붙이기 쉽고, 브라우저 `EventSource`가 **자동 재연결** 지원.
- WebSocket보다 단순, 폴링보다 효율적(연결 1개 유지, 이벤트 있을 때만 전송).

### 9.2 이벤트 채널 설계
- 클라이언트는 로그인 후 `GET /api/stream` (SSE) 구독.
- 서버는 **그룹 단위 / 사용자 단위**로 이벤트를 라우팅.
- 이벤트 종류(예):
  - `coin.updated` — 특정 쫄병 잔액 변경 (해당 쫄병에게만)
  - `product.created` / `product.updated` / `product.deleted` — 그룹 전체 쫄병에게
  - `bitto.price` — 시세 갱신 (전역 구독자 전체)
  - `order.created` — 신규 주문 발생 (해당 그룹 대장에게)
  - `request.created` — 신규 상품 신청 (해당 그룹 대장에게)
  - `request.updated` — 상품 신청 승인/거절 결과 (신청자 + 해당 그룹 대장에게)
  - `minion.approved` 등 상태 알림
- 페이로드 예:
  ```json
  { "type": "coin.updated", "balance": 1500, "delta": 100, "reason": "숙제 완료" }
  ```

### 9.3 서버 내부 전달 (확장 대비)
- 단일 서버(MVP): 인메모리 pub/sub(예: `asyncio` 큐)로 충분.
- 다중 서버로 확장 시: **Redis Pub/Sub**을 이벤트 버스로 도입해 SSE 연결이 어느 인스턴스에 붙어도 이벤트 수신. (초기엔 불필요, 확장 시점에 추가)

### 9.4 폴백
- SSE 미지원/차단 환경 대비: `EventSource` 실패 시 저빈도 롱폴링으로 자동 폴백(선택 구현).

## 15. 배포 가이드 (처음 배포하는 경우)

> 서버를 처음 올려보신다고 하셔서 단계별 방향만 정리합니다. 실제 구축은 개발 단계에서 함께 진행.

### 15.1 추천 접근 (단순 → 확장)
1. **로컬 개발**: 내 PC에서 Docker Compose로 `FastAPI + PostgreSQL + (Vue 빌드)` 실행. 먼저 완성도 올리기.
2. **1차 배포 (가장 쉬움 · 추천 시작점)**: AWS EC2 대신 **관리형 PaaS**로 시작하는 것도 방법.
   - 예: **Railway / Render / Fly.io** — DB까지 몇 클릭으로 붙고 HTTPS 자동. 서버 운영 지식이 거의 필요 없어 첫 배포에 부담이 적음.
3. **AWS EC2 직접 운영 (원래 목표)**: 학습/제어권이 필요하면 EC2로. 아래 구성.

### 15.2 AWS EC2 구성 개요
- **EC2 인스턴스**(예: t3.small, Ubuntu) 1대
- 그 위에 **Docker Compose**로:
  - `web`: FastAPI (Uvicorn/Gunicorn)
  - `db`: PostgreSQL (또는 관리형 **AWS RDS** 권장 — 백업/관리 편함)
  - `nginx`: 리버스 프록시 + 정적 파일(Vue 빌드 결과) 서빙 + **SSE 프록시 설정**(버퍼링 off)
- **도메인 + HTTPS**: 도메인 연결 후 **Let's Encrypt(certbot)** 로 무료 SSL. (SSE는 HTTPS에서 안정적)
- **환경변수/시크릿**: `.env`로 DB 접속정보·JWT 시크릿 관리(코드에 하드코딩 금지)

### 15.3 SSE 배포 시 주의점
- Nginx에서 SSE 엔드포인트는 `proxy_buffering off;`, `proxy_read_timeout` 길게 설정해야 이벤트가 즉시 흐름.
- 단일 EC2/단일 프로세스면 인메모리 pub/sub로 충분. **여러 워커/인스턴스로 확장하는 순간 Redis Pub/Sub** 필요(9.3 참고).

### 15.4 다음 단계
개발 착수 시점에 위 3가지 중 하나(추천: 첫 배포는 Railway/Render, 이후 EC2 이전)를 골라 구체적 설정 파일까지 만들어 드리겠습니다.
